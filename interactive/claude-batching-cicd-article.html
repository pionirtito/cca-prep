Batching, nightly jobs & CI/CD with Claude

An interactive walkthrough of how Claude fits into automated pipelines — the Message Batches API for bulk and scheduled work, and Claude Code headless mode for git-driven code review and CI/CD. Sourced from Anthropic's official docs.

01  Two different things called "automation"

They solve different problems and almost never appear in the same sentence by accident.

When people say "automate Claude," they mean one of two unrelated tools, and the first job is telling them apart. The Message Batches API is about volume and cost: you hand Anthropic a pile of independent requests and get them back later at half price. Claude Code headless mode is about agency in a pipeline: a single Claude that reads your repo, runs commands, edits files, and reports back inside a CI run.

The mental split Is a human waiting, and does the task need to act? Nobody waiting + independent items → Batch API. Something must read code and do things → headless Claude Code. A nightly report over 10,000 documents is batch; a PR-review bot is headless. They can even feed each other — a nightly batch job that triages issues, then a headless run that opens fixes.

02  The Message Batches API

Bundle requests, submit once, collect later, pay half.

You send up to 100,000 requests (or 256 MB, whichever comes first) as one batch. Each request is a normal Messages API call wrapped with a custom_id you choose. Anthropic processes them asynchronously — most finish within an hour, the hard ceiling is 24 hours — and everything is billed at 50% of standard prices, input and output alike, with no volume threshold to unlock the discount.

# one line per request in the batch
{
  "custom_id": "review-2026-001",        // YOUR correlation key
  "params": {
    "model": "claude-sonnet-4-6",
    "max_tokens": 1024,
    "messages": [{ "role": "user", "content": "Classify: ..." }]
  }
}

The batch starts in processing, each request runs independently — one failure never touches the others — and the status flips to ended when all are done. Results come back in no guaranteed order, which is the entire reason custom_id exists: it's how you match each response to the document it came from.

The full lifecycle, in real SDK calls

Create → poll → retrieve → resubmit only failures. Method names and fields verified against the official Python SDK and batch docs.

import anthropic
client = anthropic.Anthropic()

# 1 — CREATE: submit up to 100k requests, each with a unique custom_id
batch = client.messages.batches.create(
  requests=[
    {
      "custom_id": f"ticket-{t['id']}",        # your correlation key
      "params": {
        "model": "claude-sonnet-4-6",
        "max_tokens": 512,
        "messages": [{"role": "user", "content": classify_prompt(t)}],
      },
    }
    for t in todays_tickets        # the API rejects duplicate custom_ids
  ],
)
# batch.id → "msgbatch_01..."   batch.processing_status → "in_progress"
# batch.request_counts → {processing: N, succeeded:0, errored:0, canceled:0, expired:0}

# 2 — POLL: status flips to "ended" when every request finishes
while client.messages.batches.retrieve(batch.id).processing_status != "ended":
    time.sleep(60)                  # most batches finish < 1h; hard ceiling 24h

# 3 — RETRIEVE: results stream back in NO guaranteed order — match on custom_id
failed = []
for entry in client.messages.batches.results(batch.id):
    if entry.result.type == "succeeded":
        save(entry.custom_id, entry.result.message.content)   # full message obj
    elif entry.result.type == "errored":
        # invalid_request → fix the body before resending; else directly retryable
        failed.append(entry.custom_id)
    elif entry.result.type == "expired":
        failed.append(entry.custom_id)   # hit 24h — not billed

# 4 — RESUBMIT only the failures (no auto-retry exists), via your custom_id → data map
if failed:
    client.messages.batches.create(requests=[build(cid) for cid in failed])

Each succeeded entry holds the full message object — identical to a synchronous messages.create() response, so usage, stop_reason, and content are all there. The result types you'll branch on are succeeded, errored, canceled, and expired; an errored entry with invalid_request means the request body itself is broken (fix before resending), while other errors can be retried as-is.

Four things that bite Expiry: unfinished requests at 24h expire (and aren't billed) — collect and resubmit them. Retention: results are downloadable for 29 days from creation, then gone — persist them promptly. No retries: failed requests are not auto-retried; keep a custom_id → source-data map so you can rerun only the failures. No agentic loop: one request returns one response — if it emits tool_use, there's nowhere to run the tool and continue.

03  Sync or batch? Decide it

The exam phrases this as scenarios. One question settles every one of them.

Pick the workload — is the right lane sync or batch?
The rule Is a human staring at a spinner? Yes → synchronous, latency is the product. No → batch, take the 50%. The planted wrong answer always puts a blocking task (pre-merge check, interactive review) on batch "to save money" — a developer cannot wait up to 24 hours to merge.

SLA arithmetic

When a deadline exists, subtract the window. Need results in 30 hours, batch can take 24 → you must submit no later than 30 − 24 = 6 hours from now. For recurring loads, submit in fixed windows (say every 4 hours) so no single batch can blow its SLA if it runs the full 24.

04  Nightly & scheduled jobs

Where batch and a cron schedule meet.

"Nightly build" work splits cleanly along the same line. If the nightly task is thousands of independent analyses — classify every new support ticket, summarize every changed file, score a test corpus — that's a Batch API job kicked off by a scheduler: submit at 1 AM, collect at 6 AM, well inside the window. If the nightly task is one agent doing a multi-step job — "audit the repo for stale dependencies and open issues" — that's headless Claude Code on a schedule, not batch, because it needs the tool loop.

# nightly BULK job — scheduled batch submission
# crontab: 0 1 * * *  (1 AM daily)
submit_batch(requests=[
  {"custom_id": f"ticket-{id}", "params": {...}}
  for id in todays_new_tickets()       # 8,000 of them
])
# ... poll at 6 AM, write results to the dashboard ...

The agentic flavor of "nightly" is a headless run on a schedule. On a bare server that's a cron line; in GitHub it's a schedule trigger — both just invoke claude -p with no human present.

# nightly AGENT job — headless Claude Code on a schedule
# crontab: 0 2 * * *  (2 AM daily)
ANTHROPIC_API_KEY="$KEY" claude -p \
  "Audit the repo for stale dependencies. Open a GitHub issue
   for each one found, grouped by severity." \
  --output-format json --max-turns 8 --allowedTools "Read,Grep,Bash" \
  >> /var/log/dep-audit.log

# same thing, GitHub Actions — schedule trigger, no human present
# on: { schedule: [{ cron: "0 2 * * *" }] }
Pre-flight before any large run Dry-run a single request shape on the sync API to catch validation errors cheaply, then iterate your prompt on a 20-item sample. A bad prompt discovered at 8,000 documents is an expensive lesson — and you'll have waited hours to learn it.

05  Headless mode — the -p flag

The single most important fact for the CI/CD domain.

Run claude in a pipeline without -p and the job hangs forever waiting for interactive input — a classic CI failure. The -p (or --print) flag is the fix: Claude reads the prompt, runs its full agent loop (think → tool calls → edits), prints the result to stdout, and exits. No REPL, no approval prompts, no session. It's the only correct way to run Claude Code in automation, and it's composable with anything that runs a shell command — GitHub Actions, GitLab CI, Jenkins, a bare cron line.

# direct prompt
claude -p "Analyze this PR for security issues"

# piped stdin
cat error.log | claude -p "Diagnose this failure"

# structured, parseable output for the pipeline
claude -p "Review this PR" --output-format json

# bounded run — guardrails matter in CI
claude -p "$(cat review-prompt.txt)" \
  --output-format json --max-turns 3 --allowedTools "Read,Grep"
FlagWhy it matters in CI
-p / --printnon-interactive; without it the job hangs
--output-format jsonparseable result → post comments, gate merges
--max-turns Ncaps the agent loop — runaway runs cost real money
--allowedTools "..."least privilege — restrict what Claude can do
--model ...route cheap reviews to a smaller model
The story behind the flags A widely-cited cautionary tale: a first headless run went 47 minutes and burned $18 reformatting a file nobody touched; another silently "passed" a suite with zero real tests. The cost controls — bounded turns, restricted tools, a tight prompt, and a real CLAUDE.md — are what separate a useful gate from an expensive one.

06  Git-driven code review

The patterns the exam tests on top of the mechanism.

Use an independent instance

The same Claude session that generated code is a poor reviewer of it — it still holds its own reasoning and is unlikely to challenge its own decisions. A fresh instance, given only the diff and the review criteria without the generation context, judges the artifact instead of defending the intent. In CI this is automatic: the review job is a separate claude -p invocation with no shared history.

Don't repeat yourself across commits

On a re-review after new commits, naïvely re-running floods the PR with duplicate comments. The fix: feed the prior review results into context and instruct Claude to report only new or unresolved issues. (The official action does this differently — it buffers inline comments and classifies them via Haiku before posting, to filter out probe/test noise.)

Scope the review to stay sharp

A single pass over a 14-file PR dilutes attention — deep on a few files, shallow on the rest. Split it: one pass per file for local issues, then an integration pass for cross-file problems (type drift, circular deps). This is the multi-pass review pattern from the prompt-engineering material, applied to a diff.

CCA-F watch-list Headless = -p (not an env var, not a config key). Independent reviewer beats self-review. Structured JSON output is what lets the pipeline act (post comments, fail the build). Prior-results-in-context prevents duplicate comments. Least-privilege tools and bounded turns are the cost/safety controls.

07  The official GitHub Action

anthropics/claude-code-action@v1 — the runtime in a runner, not a thin API call.

For GitHub specifically, Anthropic ships an official action that runs the full Claude Code runtime inside a standard runner: it reads the repo, analyzes diffs, runs git, can edit code and push commits, and handles the comment/PR plumbing. The fastest setup is running /install-github-app inside Claude Code, which installs the GitHub App and wires up the secret. It auto-detects two modes:

ModeTriggerUse
Interactive@claude mention in an issue/PR/commentad-hoc: "fix the failing test in auth.ts"
Automationa prompt in the workflow YAMLruns headlessly on every matching event
# .github/workflows/claude.yml
on:
  issue_comment:        { types: [created] }   # @claude mentions
  pull_request:         { types: [opened, synchronize] }
jobs:
  claude:
    runs-on: ubuntu-latest
    permissions: { contents: write, issues: write, pull-requests: write }
    steps:
      - uses: actions/checkout@v4
      - uses: anthropics/claude-code-action@v1
        with:
          anthropic_api_key: ${{ secrets.ANTHROPIC_API_KEY }}
          prompt: "Review this PR for security and correctness"
          claude_args: "--max-turns 5 --allowedTools Read,Grep,Bash"

Key inputs (from the action's docs): prompt is the instruction — omit it on a comment event and the action responds to the trigger phrase instead; claude_args passes CLI flags straight through (--max-turns, --model, --allowedTools); anthropic_api_key is the credential (not needed on Bedrock/Vertex, which authenticate via the cloud provider).

Security, from the official docs By default only contributors with write access can trigger the action — external contributors and bots are blocked unless explicitly allowed. Setting allowed_non_write_users: "*" bypasses this and the docs flag it as a significant risk. The action also strips hidden content (HTML comments, invisible chars, image alt text) from triggers to guard against prompt injection via crafted PR/issue text. Keep the key in CI secrets — or skip the static key entirely with Workload Identity Federation (OIDC), exchanging the runner's GitHub token for a short-lived Anthropic token so there's nothing to leak or rotate.

08  The full CI/CD picture

Step through a pull-request pipeline that combines everything above.

Across these scenarios the same load-bearing ideas recur: CLAUDE.md is your CI config — review criteria, code style, and project rules defined once and obeyed on every run; -p with structured output is the bridge between Claude and the pipeline's decision logic; bounded turns and restricted tools keep cost and blast-radius in check; and an independent review instance keeps the judgment honest. Batch handles the bulk/scheduled half, the headless agent handles the act-on-the-repo half, and a scheduler stitches either into "nightly."
